# Paymenter API Guide

This document describes how to work with the Paymenter REST API. It covers authentication, available endpoints, and examples for using the API.

## Base URL

All API routes are located under the `/api` prefix. The following endpoints are available:

- `POST /api/oauth/token` – issue an OAuth token for user based access
- `GET  /api/me` – retrieve the authenticated user's details
- `*   /api/v1/admin/...` – admin focused API endpoints (described below)

## Authentication

Admin endpoints require a bearer token generated from the **API Keys** page in the admin panel. Include the token in the `Authorization` header when performing requests:

```http
Authorization: Bearer <your-api-token>
```

OAuth access tokens can be generated by sending a `POST` request to `/api/oauth/token` with the required credentials.

## Pagination, Sorting and Filtering

List endpoints accept query parameters powered by [spatie/laravel-query-builder](https://github.com/spatie/laravel-query-builder). Common parameters include:

- `per_page` – number of items per page (default `15`)
- `page` – page number to return
- `filter[*]` – filter by an attribute (e.g. `filter[email]=example@example.com`)
- `sort` – comma separated list of attributes to sort by
- `include` – load relationships (see each controller for available includes)

Responses follow the [JSON:API](https://jsonapi.org/) specification.

## Admin Endpoints

The following resources are available under `/api/v1/admin`:

| Resource | Endpoints | Permissions |
| --- | --- | --- |
| **Users** | `GET /users`, `POST /users`, `GET /users/{id}`, `PUT/PATCH /users/{id}`, `DELETE /users/{id}` | `users.view`, `users.create`, `users.update`, `users.delete` |
| **Orders** | `GET /orders`, `POST /orders`, `GET /orders/{id}`, `PUT/PATCH /orders/{id}`, `DELETE /orders/{id}` | `orders.view`, `orders.create`, `orders.update`, `orders.delete` |
| **Services** | `GET /services`, `POST /services`, `GET /services/{id}`, `PUT/PATCH /services/{id}`, `DELETE /services/{id}` | `services.view`, `services.create`, `services.update`, `services.delete` |
| **Invoices** | `GET /invoices`, `POST /invoices`, `GET /invoices/{id}`, `PUT/PATCH /invoices/{id}`, `DELETE /invoices/{id}` | `invoices.view`, `invoices.create`, `invoices.update`, `invoices.delete` |
| **Tickets** | `GET /tickets`, `POST /tickets`, `GET /tickets/{id}`, `PUT/PATCH /tickets/{id}`, `DELETE /tickets/{id}` | `tickets.view`, `tickets.create`, `tickets.update`, `tickets.delete` |
| **Ticket Messages** | `GET /ticket-messages`, `POST /ticket-messages`, `GET /ticket-messages/{id}`, `DELETE /ticket-messages/{id}` | `ticket_messages.view`, `ticket_messages.create`, `ticket_messages.delete` |

Each endpoint returns a JSON response describing the resource. Relationships can be included with the `include` query parameter if the authenticated token has permission to view them.

### Example: Listing Users

```bash
curl -H "Authorization: Bearer <token>" \
     'https://your-paymenter.example/api/v1/admin/users?per_page=20&include=role'
```

### Example: Creating an Invoice

```bash
curl -X POST 'https://your-paymenter.example/api/v1/admin/invoices' \
     -H 'Authorization: Bearer <token>' \
     -H 'Content-Type: application/json' \
     -d '{
         "user_id": 1,
         "currency_code": "USD",
         "due_at": "2024-12-01",
         "status": "pending"
     }'
```

### Response Format

Resources use dedicated JSON:API resources found under `app/Http/Resources`. For example, `UserResource` exposes a user's basic fields along with optional relationships such as `orders` or `services`.

## Permissions

An API token can be restricted to specific actions. Available permissions are defined in `config/permissions.php` under the `api.admin` section. When a request attempts to include a relationship or perform an action without the matching permission, a `403` response is returned.

## Error Handling

The API uses standard HTTP status codes. `400` level responses indicate validation failures or permission issues. `401` is returned for invalid or missing API tokens.

## Further Reading

See the code under `routes/api.php` and `app/Http/Controllers/Api/Admin` for exact behaviour of each endpoint. The repository also contains automated tools for generating an OpenAPI specification which can be used to explore the API in detail.
